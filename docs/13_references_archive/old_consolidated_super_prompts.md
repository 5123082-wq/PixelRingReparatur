# Consolidated Super Prompts
## Stitch + Antigravity
### AI-first multilingual one-stop service company for signs, repairs, installation and branding

---

## 1. SUPER PROMPT FOR STITCH

Design a high-fidelity multilingual website for a modern AI-first one-stop service company that handles sign repair, sign installation, dismantling, light advertising, new sign production, branding, window graphics, vehicle branding and related advertising-service tasks.

Critical business requirement: the website must NOT look like a contractor marketplace, gig platform, listing directory, or lead marketplace. It must feel like a real service company with one entry point, one accountable process, one service brand and one consistent customer journey.

### Brand perception goals
The user must instantly feel that:
- this is a real service company;
- this is not a marketplace of random masters;
- they do not need to search for a contractor themselves;
- they can start a request very quickly;
- AI helps with intake, but a real specialist continues the process;
- the company remains responsible for the service process.

### Core business concept
The company reduces friction and cost by removing the classic manager layer at the first stage. Instead of long calls with a sales manager, the user can describe the task via text, photo, messenger or voice. AI helps structure the request, then the company assigns a relevant specialist.

### Languages
The design must support 6 languages from the start:
- German (primary language)
- English
- Russian
- Turkish
- Polish
- Arabic

Design must support:
- multilingual text expansion;
- long German labels;
- Russian and Polish text length;
- RTL-ready layout for Arabic.

### Design positioning
The design should feel like:
- modern service operations platform;
- high-trust B2B/B2SMB service website;
- clean, structured, confident, practical;
- technology-enabled but not “AI gimmick driven”.

The design must NOT feel like:
- contractor directory;
- marketplace cards;
- ratings marketplace;
- chaotic local services portal;
- generic creative agency site;
- startup hype landing page.

### Visual direction
Use a clean, structured, high-trust service design language with:
- strong typography;
- clear hierarchy;
- visible CTA buttons;
- generous spacing;
- card-based section organization;
- operational clarity;
- realistic service-oriented visuals;
- mobile-first thinking.

Avoid:
- clutter;
- noisy visuals;
- excessive gradients just for trendiness;
- overdecorated AI aesthetics;
- contractor portraits as the primary proof mechanic.

### Primary homepage goals
The homepage must:
- explain the service model within 3 seconds;
- show that this is one service company;
- create trust before asking for action;
- give multiple easy ways to start;
- prevent “marketplace confusion”;
- make mobile usage effortless.

### Required homepage sections
1. Hero section
2. Trust block (“Why trust us”)
3. Ways to submit a task
4. Task scenarios / problem cards
5. How it works (3 steps)
6. Why it’s easier and more efficient
7. Proof / examples / case section
8. What happens after submission
9. FAQ
10. Final CTA block

### Homepage content logic
The first screen should communicate:
- one service company;
- easy start;
- AI-assisted intake;
- human specialist after intake;
- no need to search contractors manually.

The homepage must include clear entry paths for:
- start request;
- upload photo;
- continue in WhatsApp;
- continue in Telegram;
- voice / phone path.

### Required internal pages
Design these pages too:
- Homepage
- How it works
- Services hub
- Service page template
- Business / multi-location page
- Cases / examples page
- Contact page
- Start request / intake flow
- Thank-you / confirmation state
- Legal page template

### Service page requirements
Each service page should:
- explain the task simply;
- use scenario language, not technical overload;
- show what the user can send to start;
- include FAQ and CTA;
- support photo-first entry.

### How-it-works page requirements
This page must clearly explain:
- what AI does;
- what the company does;
- what the specialist does;
- how the handoff works;
- that the request stays inside one service process.

### Intake flow requirements
The intake must not look like a long form.
It must look like a guided service intake.

It should support:
- text-first path;
- photo-first path;
- messenger continuation;
- voice path;
- step-by-step flow;
- clear progress;
- soft reassurance copy.

### Thank-you state requirements
The confirmation state must explain:
- the request was received;
- the core data was saved;
- a specialist will continue;
- the user does not need to start over;
- they can still add details if needed.

### Mobile-first requirements
Mobile is the primary use case.
Design must include:
- strong above-the-fold CTA;
- sticky CTA on mobile;
- easy one-thumb actions;
- fast photo upload placement;
- visible messenger buttons;
- easy language switching.

### Critical trust instructions
Design explicit trust architecture. The user must never feel:
- “this is just another platform to find random workers”;
- “my number will be passed around to unknown people”;
- “I’m talking only to a bot”.

The design must constantly reinforce:
- service-company identity;
- structured process;
- company accountability;
- AI as intake helper, not final actor.

### What to avoid completely
Do not create:
- specialist listing grids;
- contractor rating widgets;
- compare-provider interfaces;
- “choose your master” patterns;
- marketplace recommendation cards;
- freelance platform vibes.

### Final output expected from Stitch
Generate a high-fidelity design package containing:
- Homepage
- Service page template
- How-it-works page
- Business page
- Cases page
- Contact page
- Intake flow screens
- Thank-you state
- Mobile variants
- Multilingual-safe design system
- RTL-aware Arabic-compatible layouts

### Final self-check for Stitch
Before finalizing, review the design and improve it until all are true:
- It feels like one service company.
- It does not feel like a marketplace.
- Trust appears before heavy conversion pressure.
- The next action is obvious.
- The intake feels easy.
- Mobile feels primary.
- All 6 languages can fit the layout.
- Arabic can be supported safely.

## 2. SUPER PROMPT FOR ANTIGRAVITY

Build a production-ready multilingual web application for a modern AI-first one-stop service company that handles sign repair, installation, dismantling, light advertising, new signs, branding, window graphics, vehicle branding and related service requests.

Critical business rule: this product must NOT behave like a contractor marketplace. It must behave like a real service company with one intake layer, one lead model, one customer-facing brand, one service process and one accountable handoff flow.

### Product model
Users should be able to:
- understand the service instantly;
- start a request quickly;
- submit via text, photo, messenger or voice;
- interact with AI-assisted intake;
- know that a human specialist will continue;
- stay inside one accountable service system.

### Languages
Support 6 languages from the start:
- German (primary, canonical)
- English
- Russian
- Turkish
- Polish
- Arabic

Requirements:
- locale-aware routing;
- translatable content architecture;
- locale-specific UI strings;
- multilingual intake and AI messages;
- preserved preferred language in lead model;
- RTL-aware Arabic support.

### Core public pages to build
- Homepage
- How it works
- Services hub
- Service page template(s)
- Business / multi-location page
- Cases page
- Contact page
- Start request / intake
- Thank-you state
- Legal pages

### Core functional layers to build
1. Multilingual website shell
2. Section-based page system
3. Guided intake flow
4. AI-assisted intake orchestration
5. Photo upload path
6. WhatsApp / Telegram continuation paths
7. Voice / phone entry path or architecture-ready placeholder
8. Unified lead model / CRM storage
9. Admin dashboard
10. Status and handoff logic
11. Assignment-ready internal data model
12. Analytics instrumentation
13. Security and privacy baseline

### Public UX requirements
The site must:
- feel trust-first;
- make the next step obvious;
- support mobile as the primary mode;
- support photo-first entry naturally;
- support messenger continuation naturally;
- preserve one-stop service perception;
- avoid contractor directory patterns entirely.

### Intake requirements
Build the intake as guided service intake, not a long form.

It must support:
- scenario-first start;
- quick path;
- photo-first path;
- progressive steps;
- preferred contact method;
- preferred language;
- clear confirmation;
- explicit human handoff state.

### AI requirements
The AI layer must:
- help structure the request;
- ask one useful question at a time;
- support all 6 languages;
- produce structured lead data;
- produce a concise summary;
- support fallback paths;
- trigger handoff when minimum data is reached or user requests a human.

The AI layer must NOT:
- act like a marketplace agent;
- say it will “find a contractor”;
- expose internal partner mechanics;
- fake precise prices;
- pretend it replaces the human specialist.

### Lead model requirements
Every request, regardless of channel, must normalize into a unified lead model.

Each lead should support:
- source channel;
- intake channel;
- preferred language;
- service scenario;
- urgency;
- address / geo data;
- contact details;
- attachments;
- AI transcript and summary;
- current status;
- handoff state;
- assignment-ready fields.

### Admin dashboard requirements
Build a secure internal admin dashboard where operators can:
- view new leads;
- filter by status, language, urgency, service and source;
- view attachments;
- see transcript / summary;
- assign owner;
- update status;
- track handoff;
- prepare specialist assignment.

This dashboard must not expose more data than needed per role.

### Internal assignment model requirements
Create an internal assignment-ready structure for specialists/partners.

It must support:
- service capability matching;
- geography matching;
- language matching;
- availability;
- quality status;
- assignment ownership.

Important: this is internal only. Do NOT expose specialists publicly as a marketplace directory.

### Analytics requirements
Instrument the product with privacy-aware analytics.
Track at minimum:
- page_view
- CTA_click
- intake_started
- intake_step_completed
- photo_uploaded
- messenger_handoff_clicked
- voice_entry_started
- intake_completed
- handoff_triggered
- lead_created
- language_selected

Track dimensions such as:
- language
- source
- device
- page
- scenario
- channel

Do not leak sensitive lead content into general analytics events.

### Security and privacy requirements
Use security-by-design and privacy-by-design defaults.

Mandatory requirements:
- no hardcoded secrets;
- server-side authorization for protected resources;
- role-based access control;
- secure file upload handling;
- no public exposure of private files;
- minimal PII exposure;
- consent-aware forms and flows;
- secure handling of API keys;
- audit logging for privileged actions;
- avoid unsafe debug output.

### Anti-marketplace review requirement
Continuously review the product and refactor anything that creates a marketplace perception.

Remove or avoid:
- contractor cards;
- listing-like components;
- compare-provider logic;
- wording like “find a master” or “choose a contractor”;
- public partner mechanics.

The product must always feel like one accountable service company.

### Technical implementation style
Use a modular, maintainable, reusable architecture with:
- section-based public pages;
- reusable page templates;
- locale-aware routing;
- reusable CTA system;
- reusable intake components;
- explicit state management for handoff;
- audit-friendly internal actions.

### Build order
Build in this order:
1. Multilingual shell and routing
2. Homepage and core public pages
3. Guided intake and thank-you flow
4. Photo upload path
5. AI intake orchestration
6. Lead model and storage
7. Admin dashboard and statuses
8. Messaging continuation
9. Assignment-ready internal model
10. Analytics and security hardening

### Required review after each phase
After each major build phase, review and improve against these questions:
- Does it still feel like a service company?
- Is the next action obvious?
- Is the intake too complex?
- Is multilingual support intact?
- Is mobile still strong?
- Is the AI layer safe and useful?
- Is any internal partner logic leaking into public UI?
- Are security and privacy guardrails preserved?

### Launch-ready definition
The build can be considered launch-ready only if:
- the public site works in all 6 languages;
- the homepage is trust-first;
- intake works end-to-end;
- photo path works;
- AI handoff works;
- leads are stored and visible in admin;
- statuses and ownership are clear;
- no critical marketplace cues remain;
- no critical privacy/security issues remain in core flows.

### Final output expected from Antigravity
A production-ready MVP that includes:
- multilingual public site
- trust-first homepage
- core service pages
- how-it-works page
- business page
- cases page
- contact page
- AI-assisted intake flow
- photo-first path
- messenger continuation paths
- thank-you state
- unified lead model
- secure admin dashboard
- status / handoff logic
- analytics basics
- privacy/security baseline
