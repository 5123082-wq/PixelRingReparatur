# Документ 8
## MVP Delivery Plan / Backlog / Launch Roadmap
### AI-first multilingual one-stop service company for signs, repairs, installation and branding

---

## 1. Назначение документа
Этот документ переводит всю предыдущую стратегию в управляемый план реализации.

Он описывает:
- scope MVP;

- этапы реализации в Antigravity;

---

## 2. Принцип запуска

### 2.1. Что нельзя делать
Нельзя пытаться запустить “идеальную полную платформу” в первой версии.

### 2.2. Что нужно делать
Нужно запускать **управляемый MVP**, который уже:
- не выглядит как маркетплейс;
- умеет принимать заявки;
- поддерживает 6 языков;
- умеет вести пользователя к заявке;
- поддерживает AI intake;
- передает задачу человеку;
- имеет рабочий операционный контур.

### 2.3. Формула MVP
**Минимум функций, достаточных для доверия, intake, handoff и управления заявками.**

---

## 3. Что входит в MVP

### 3.1. Продуктовый scope MVP
В MVP входят:
- мультиязычный сайт на 6 языках;
- главная страница;
- страницы услуг;
- страница “Как это работает”;
- страница “Для бизнеса”;
- кейсы / примеры задач;
- контакты;
- legal pages;
- AI-assisted intake;
- фото-first вход;
- WhatsApp / Telegram handoff;
- phone / voice entry;
- базовый CRM / lead storage;
- admin view;
- assignment logic;
- базовая аналитика.

### 3.2. Что критично для MVP
- trust-first brand facade;
- no-marketplace perception;
- one-stop service language;
- 6-language UX;
- human handoff clarity;
- заявки не теряются.
- управляемая админ-панель без code-only публикации контента.

---

## 4. Что НЕ входит в MVP

### 4.1. Отложить на phase 2+
- полноценный клиентский личный кабинет;
- сложный self-service tracking portal;
- расширенная партнерская панель;
- сложные автоматические ценообразования;
- тяжелая BI-аналитика;
- отдельные гео-кластеры по всем районам / городам;
- сложный account management для корпораций;
- deep automation for scheduling / dispatch.

### 4.2. Почему
Потому что first priority — не complexity, а:
- доверие;
- intake;
- handoff;
- управляемость;
- реальный запуск.

---

## 5. MVP goals

### 5.1. Главные цели MVP
1. Проверить, что бренд не воспринимается как marketplace.
2. Проверить, что люди готовы оставлять заявки через AI-first flow.
3. Проверить, что multilingual UX реально повышает конверсию.
4. Проверить, что handoff к человеку проходит без потерь.
5. Проверить, что операционный контур выдерживает входящий поток.

### 5.2. Непродуктовая цель
Сделать систему достаточно простой для быстрого тестирования и улучшения.

## 11. MVP backlog priority model

### 11.1. P0 — absolutely required before launch
- multilingual shell
- homepage
- service pages core
- intake flow
- photo upload
- AI layer basic
- human handoff logic
- admin lead view
- lead statuses
- contact and legal pages
- analytics basics
- admin foundation security baseline (`audit + CSRF + role split`) and stable CMS CRUD for core public content.
- CMS media boundary enforcement (public media vs private customer attachments) with upload validation and audit baseline.

### 11.2. P1 — important but can finalize near launch
- cases page
- business page
- improved FAQ
- messenger continuation refinement
- SLA alerts
- better admin filters
- Content Core expansion: revision history, workflow states beyond draft/published, signed preview, scheduled publishing, managed navigation.
- Assets + Forms expansion: media governance metadata/reporting, image derivative pipeline, configurable request/contact form schemas with server-side validation.
- Delivery + Integrations hardening: event/webhook outbox flow, deterministic cache revalidation, adapter isolation, retry/idempotency policy for external channels.

### 11.3. P2 — post-launch improvements
- advanced voice handling
- deeper dashboards
- geo expansion pages
- business account enhancements
- more advanced partner tooling
- hardening + release maturity: named admin users with granular RBAC, MFA/step-up, distributed rate limiting, backup/restore drills, incident runbooks, and stricter release gates.

## 12. Acceptance criteria by phase

### 12.1. Design phase accepted if
- дизайн не выглядит как marketplace;
- CTA clear;
- key flows cover mobile;
- multilingual layout holds;
- trust system встроен во все ключевые экраны.

### 12.2. Build foundation accepted if
- сайт открывается на 6 языках;
- routing стабилен;
- ключевые templates готовы;
- language switching не ломает UX.

### 12.3. Intake phase accepted if
- пользователь может начать обращение меньше чем за минуту;
- есть photo-first path;
- есть quick path;
- есть clear thank-you state.

### 12.4. AI phase accepted if
- AI не звучит как marketplace bot;
- AI умеет собрать минимум данных;
- AI правильно инициирует handoff;
- диалоги работают на 6 языках.

### 12.5. Operations phase accepted if
- лиды не теряются;
- статусы управляемы;
- есть owner;
- assignment работает;
- admin может контролировать процесс.

### 12.6. Launch accepted if
- end-to-end flow проходит без разрывов;
- UX trust intact;
- multilingual QA пройден;
- analytics пишется;
- operators готовы.

## 16. Risk register

### 16.1. Product risks
- сайт выглядит как marketplace;
- AI воспринимается как barrier, а не helper;
- слишком много шагов в intake;
- слабая multilingual execution;
- слабый trust layer.

### 16.2. Operational risks
- leads теряются;
- assignment медленный;
- language mismatch;
- poor partner response;
- no clear owner.

### 16.3. Delivery risks
- слишком большой MVP;
- желание сразу сделать все;
- разрыв между дизайном и реализацией;
- слабая связка Stitch → Antigravity.

### 16.4. Mitigation logic
- scope discipline;
- P0/P1/P2 split;
- design review against trust goals;
- end-to-end flow testing;
- operational rehearsal before launch.

## 18. Success criteria for MVP launch

MVP можно считать успешным, если:
- пользователь понимает модель сервиса сразу;
- сайт не воспринимается как marketplace;
- заявки поступают и не теряются;
- 6 языков реально работают;
- AI помогает, а не мешает;
- handoff к человеку прозрачен;
- операции управляемы;
- продукт можно улучшать без полного переделывания.
